# Spec: Tailored Résumé — Provenance Pipeline

Implements `tailored-resume-blueprint.md` rev 3 (grilled READY 2026-07-13).
Self-contained: a fresh agent can implement from this file alone. Repo root for
all paths below: `dashboard/`.

## Goal

From an existing job row, generate a tailored résumé section-by-section with
human approval, where every generated bullet is mechanically traceable to
human-confirmed evidence, exported as .docx, with a before/after skill-match
score. Honesty is enforced by code (claim-id validation, digit-diff, entailment
judge), not by prompt hope — research showed every commercial tool fabricates
metrics and prompt-only constraints fail.

## Why (background the implementer needs)

- **Evidence-closed generation:** the model only ever sees human-confirmed
  claims + the section's own original text. Every bullet must cite claim ids.
  This is the product's differentiator; nothing on the market has it.
- **The base résumé lives as a PDF** in Supabase Storage (`cvs/<id>.pdf`). The
  repo has no local PDF-text extraction, so a one-time LLM transcription
  (human-correctable) becomes the canonical `full_text` — the single ground
  truth for splitting, evidence slicing, and verbatim carryover.
- **Vercel Hobby cap:** 12 functions; repo has 8 files in `api/` (each counts,
  helpers included). This feature adds EXACTLY ONE: `api/tailor.js`. All other
  server code goes in `api-lib/` (outside `api/`, costs no slot).
- **`matchJob()` score is a Goodhart target** — display only, never in a prompt.

## Scope

**In:** SQL migration; `sourceStore.download`; `api/tailor.js` dispatcher
(4 actions); `api-lib/tailor/{prompts,judge}.js`; `src/tailor/{anchor,provenance,docx}.js`;
`TailorScreen` UI + trigger; pill-claim minting; before/after score; tests;
notebook-verify.

**Out (explicitly):** cover letters; CV list/management UI; in-app `full_text`
editor (corrections via Supabase console); project-template synthesis (companion
plan — this feature READS `project_template` rows, never writes them); layout
reuse of the original PDF; multiple résumé styles; `tailor_session` persistence
(approval progress + pill confirmations are client state, lost on refresh,
re-confirmable); streaming responses; eval automation (the `tailor_log` trail is
built now, the eval harness later); any change to `api/file.js` (CV retrieval
stays off the public route — DECISIONS.md 2026-07-13).

## Data schemas

### SQL migration (manual, Supabase editor — step 0)

```sql
alter table cv add column full_text text;
alter table cv add column sections jsonb;

create table project_template (
  id bigint generated by default as identity primary key,
  name text not null,
  claims jsonb not null default '[]'::jsonb,
  created_at timestamptz not null default now()
);

create table tailor_log (
  id bigint generated by default as identity primary key,
  job_id text,
  cv_id bigint,
  section text,
  guard text not null,        -- 'hard' | 'digit' | 'judge' | 'judge-error'
  pass boolean not null,
  objection text,
  payload jsonb,              -- { bullets, evidence: [{id, text}] }
  created_at timestamptz not null default now()
);

-- RLS (tables created via raw SQL default to RLS OFF — do not skip):
alter table project_template enable row level security;
create policy "anon read templates" on project_template for select to anon using (true);
-- no anon write policies: templates are authored via the service role/console only.
alter table tailor_log enable row level security;
-- NO anon policies at all: tailor_log is written by the service role (api/tailor.js)
-- and read only via console/service role. The browser never touches it.
```

### Shapes (JS)

```yaml
cv.sections:            # written only by action=split
  full_text_hash: <sha256 hex of cv.full_text exactly as stored>
  sections: [{name: string, start: int, end: int}]   # [start,end) into full_text

claim:                  # project_template.claims[] element
  id: string            # GLOBALLY unique across ALL templates, e.g. "jp-1"
  text: string          # atomic, verifiable, e.g. "Built X using Y"
  skills: [string]      # canonical skill names, stamped at synthesis time
# Global uniqueness is enforced loudly: whenever the server loads templates
# (generate/revise step 3a), a duplicate id across templates → 500
# {error:'duplicate claim id: <id> in templates <a>,<b>'}. Collisions are
# authoring mistakes to catch, not cases to support (grill spec-r1 HIGH).

pill_claim:             # minted client-side on pill confirm (component 9)
  id: "pill-<canonical>"        # e.g. "pill-Docker"
  text: "Has skill: <canonical> (self-confirmed <YYYY-MM-DD>)"
  source: "pill"

orig_claim:             # minted SERVER-side per generate/revise call
  id: "orig-<sectionName>"
  text: <section slice of full_text>
  source: "orig"

bullet:                 # generation output unit
  text: string
  claim_ids: [string]   # non-empty, every id ∈ evidence set
```

**Evidence set per generate/revise call** = selected template claims (by
`claimIds`) ∪ `pillClaims` (request body) ∪ the section's orig-claim (server-
minted). The prompt contains exactly these plus JD required skills — nothing else.

export function buildSplitBody({ fullText })

`SYSTEM_RULES` (verbatim, part of the prefix):

```
You tailor résumé sections. Rules, absolute:
- Use ONLY the evidence provided (claims and the section's original text).
- Every bullet MUST list the claim_ids it draws from. No claim_id, no bullet.
- Never introduce a skill, metric, number, employer, title, or capability that
  is not in the evidence. Rephrasing and reordering are allowed; inventing is not.

```

Unit-testable assertion (Goodhart invariant): no prompt string produced by any
builder contains the substring `score` case-insensitively or any `matchJob`

### C3 — `api-lib/tailor/judge.js` (server-only)

```js
// → {pass: boolean, objection: string|null}. NEVER throws.
// Infrastructure failure (fetch throw, non-200, malformed response) →
// {pass:false, objection:'judge unavailable', judgeError:true}.
export async function judge({ bullets, evidence, apiKey })
```
One Haiku call, `temperature: 0`, forced tool `verdict` with schema
`{pass: boolean, objection: {type:['string','null']}}`. Judge user text:
evidence list (id + text) + bullets, question: "Is every factual statement in
these bullets entailed by the evidence? Rephrasing is fine; any new fact,
number, skill, or capability is not."

### C4 — `src/tailor/anchor.js` (pure, shared-safe)

```js
// Locate each section's verbatim quotes in fullText and return offsets.
// quotes: [{name, heading_quote, first_line_quote}] (LLM output, order = document order)
// → {ok:true, sections:[{name,start,end}]} |
//   {ok:false, gap:{section, quote}}   // ONE shape for every failure mode:
//   heading-quote not found AND first-line-quote outside its span both report
//   the offending section name + the quote that failed to anchor. This exact
//   object is the split action's 422 `gap` payload.
export function anchorSections(fullText, quotes)
```
- Locate `heading_quote` by exact `indexOf` (search forward from previous
  section's end); miss → fuzzy: sliding window, normalized similarity ≥ 0.85
  (character-level, whitespace-collapsed); still miss → `ok:false,
  gap:{section, quote: heading_quote}`.
- Section span = its heading start → next section's heading start (last section
  → `fullText.length`).
- **Coverage rule (falsifiable):** spans tile `fullText` by construction
  (heading → next heading), so tiling itself proves nothing. The real check:
  **each section's `first_line_quote` must anchor (exact, then fuzzy ≥ 0.85)
  INSIDE that section's computed span** — a model that omits a heading leaves
  some section's first line inside a neighbor's span and the check fires.
  Any first-line miss → `ok:false, gap:{section, quote}`.
- Text BEFORE the first heading (name block, contact info) is section
  `{name:'_header', start:0, ...}` — auto-prepended, carried over verbatim,
  never generated against, exempt from the first-line check (it has no quotes).

```js
export async function sha256Hex(text)  // canonical hash for full_text staleness
```
ONE implementation for both runtimes: `globalThis.crypto.subtle.digest('SHA-256',
new TextEncoder().encode(text))` → hex. Available in browsers AND Vercel/Node
≥ 18 — no `node:crypto` import anywhere in `src/` (Vite cannot bundle it).

### C5 — `src/tailor/provenance.js` (pure, shared-safe)

```js
// HARD guard. → {ok:true} | {ok:false, reason:string}
// Fails when: bullets missing/empty; any bullet with empty claim_ids; any
// claim_id not in evidenceIds.
export function validateClaimIds(bullets, evidenceIds)

// SOFT guard. → {ok:true} | {ok:false, digits:[string]}
// Extract number tokens from bullet texts (regex /\d[\d,.]*%?/g). Normalize
// each token: strip commas, then strip trailing '.' and ',' (sentence-final
// "40." → "40"; "3.11." → "3.11"). Evidence side: tokenize every evidence
// text with the SAME regex, normalize each token identically, build a Set —
// fail if any bullet token is absent from that Set (set membership, NOT
// substring search: evidence "1,405" must not legitimize a fabricated "40").
// Accepted token-split cases (documented, not bugs): "24/7" splits
// to "24" and "7"; ranges "2019–2023" split to "2019" and "2023" — each part
// must independently appear in evidence. Years inside orig-claim text pass by
// construction.
export function digitDiff(bullets, evidenceTexts)
```

### C6 — `api/tailor.js` (the ONLY new file in `api/` — slot 9/12)

`POST` only (405 otherwise). Body `{action, ...}`. Missing env → 500. Unknown
action → 400. Supabase service-role client per `file.js` pattern.

| action | body | 200 response |
|---|---|---|
| `transcribe` | `{cvId, overwrite?}` | `{fullText}` |
| `split` | `{cvId}` | `{sections:[{name,start,end}]}` |
| `generate` | `{jobId, cvId, sectionName, claimIds:[], pillClaims:[]}` | `{bullets, verified, objection?}` |
| `revise` | generate body + `{note, priorBullets}` | `{bullets, verified, objection?}` |

**transcribe:** load cv row (404 if missing); IF `cv.full_text` non-null AND
`overwrite !== true` → 409 `{error:'full_text exists; pass overwrite:true'}`;
`download(supabase, cv.pdf_path)` (404 if no path or bytes null); Anthropic
transcribe call; persist `full_text`; **null out `cv.sections`** (a new
transcript invalidates any split); return.

**split:** cv row + `full_text` required (404); Anthropic split call →
`anchorSections`; `ok:false` → 422 `{error:'split coverage failed', gap}`;
persist `cv.sections = {full_text_hash: await sha256Hex(full_text), sections}`
(overwrite allowed — re-derivable); return sections.

**generate/revise — guard pipeline (exact order):**
1. Load cv + `cv.sections` (404 if either absent). **Staleness:**
   `await sha256Hex(cv.full_text) !== cv.sections.full_text_hash` → 409
   `{error:'stale split — re-run split'}`.
2. Load job row + its `skill` rows: `select canonical from skill where job_id =
   jobId and requirement = 'required'` → `jdSkills` (deduped canonical list).
   Unknown `jobId` (no job row) → 404. `sectionName` not in sections → 404.
   Slice orig-claim text from `full_text`.
3. Resolve evidence: (3a) load ALL `project_template` rows; assert global
   claim-id uniqueness — duplicate → 500 naming the id and templates. Then
   template claims for `claimIds` (unknown id → 400) ∪ `pillClaims` (validate
   shape: id starts `pill-`, has text) ∪ orig-claim.
4. Anthropic generate call (prefix + suffix).
5. **Hard guard** `validateClaimIds` → fail: ONE retry (objection in suffix) →
   fail again: 422 `{error:'generation failed provenance', reason}`; log both
   attempts (`guard:'hard'`).
6. **Soft guards** on the hard-passing result: `digitDiff` first —
   **short-circuit: on digit failure the judge does not run** (its objection
   drives the retry) — then `judge(...)`. First soft failure: ONE retry
   (objection in suffix). The retry result re-runs the FULL chain (hard →
   digit → judge); any failure on the retry is final: hard → 422 (per step 5's
   "fail again"), soft → 200 `{bullets, verified:false, objection}`. All
   clean: 200 `{bullets, verified:true}`.
   **Retry budget (hard cap): max 3 generation calls per request** — initial +
   at most one hard retry + at most one soft retry. The judge runs at most
   twice. Objection on retry = the failing guard's objection only (hard:
   validator reason; digit: the foreign tokens; judge: its objection text).
7. `judge.judgeError === true` → 200 `{bullets, verified:false,
   objection:'judge unavailable'}` — no retry (infra, not content).
8. **Every** guard/judge evaluation appends one `tailor_log` row
   (`guard: 'hard'|'digit'|'judge'|'judge-error'`, `pass`, `objection`,
   `payload: {bullets, evidence:[{id,text}]}`) — best-effort, log failure never
   fails the request.
9. Anthropic call itself fails (non-200 / fetch throw) → 502
   `{error:'generation failed'}` (transcribe/split too). Judge failure is NEVER
   502 (step 7).

No response ever contains: API keys, bucket URLs, signed URLs, service-role
material, or prompt text.

### C8 — UI (`src/tailor/TailorScreen.jsx` + `SectionCard.jsx`, wired in `App.jsx`)

- **Trigger:** "Create a résumé" button in the expanded job row (next to the
  existing compare/screenshot actions) → renders `TailorScreen` for that job +
  the selected cv (v1: most recent cv row with non-null `full_text` and
  `sections`; none → the screen shows setup instructions naming transcribe/split).
- **Layout:** left pane = JD screenshot (`GET /api/file?kind=screenshot&id=`,
  existing lightbox pattern); right pane = section flow.
- **Before the loop:** skill-gap pills — compute `jdRequired = new
  Set(job.skills.filter(r=>r.requirement==='required').map(s=>s.canonical))`
  minus (cv canonical set ∪ all template-claim `skills`); each pill Confirm →
  mint pill_claim (shape above); **Reject → the skill is dropped entirely: no
  claim minted, excluded from the after-score set, never re-prompted this
  session.** Then per-project claim checklist (all template claims listed,
  pre-checked; uncheck freely). **Checklist data source:** the browser reads
  `project_template` directly via the existing anon supabase client
  (`src/supabase.js` house pattern) — RLS grants anon select-only.
- **Section loop, in `cv.sections` order** (skip `_header` — always carryover):
  states `pending → generating → review(verified|unverified badge) → approved`.
  Review shows bullets; per-bullet tap highlights cited claims (from
  `claim_ids`); actions: Approve / revision note (regenerates via `revise`,
  same-priority reject affordance — no rubber-stamp bias) / Skip (stays
  carryover).
- **Unverified badge:** amber "unverified — not confirmed against your claims"
  chip + the objection text. 422 → red error state + Retry.
- **Early exit:** "Ready — download my résumé" always visible. Assembly: fetch
  `cv.full_text` + `cv.sections` in ONE select; assert `await
  sha256Hex(full_text) === sections.full_text_hash` — mismatch → export blocked, message "résumé
  text changed — re-run split", no file produced. Approved sections use their
  bullets; everything else carries over `full_text.slice(start, end)` verbatim.
- **Score:** `before = matchJob(job, cvSet)`; `after = matchJob(job, new
  Set([...cvSet, ...confirmedPillCanonicals]))`. Render both next to download.
  Score is display-only (never sent to `/api/tailor`).
